# JSON Web Tokens (JWT): The Senior Engineer's Deep-Dive Study Guide

This guide is designed for preparing for L5+ (Senior/Staff) systems design and architecture interviews at companies like Google, Meta, Amazon, and Netflix. It covers the core mechanics of JWT, threat modeling, stateful vs. stateless tradeoffs, distributed revocation patterns, and advanced system design scenarios.

---

## 1. Core Architecture & Mechanics

### What is a JSON Web Token?
A JSON Web Token (JWT) is an open standard ([RFC 7519](https://tools.ietf.org/html/rfc7519)) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object.

A JWT is composed of three parts separated by dots (`.`): `Header.Payload.Signature`.

```
┌─────────────────────────────────────────────────────────────┐
│                       JWT STRUCTURE                         │
├─────────────────────────────────────────────────────────────┤
│  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9                       │ <-- Header (Base64URL)
│  .                                                          │
│  eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwi...    │ <-- Payload (Base64URL)
│  .                                                          │
│  SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c                │ <-- Signature
└─────────────────────────────────────────────────────────────┘
```

#### A. The Header
Typically consists of two parts: the type of the token (`JWT`) and the signing algorithm being used (e.g., `HS256`, `RS256`).
```json
{
  "alg": "RS256",
  "typ": "JWT"
}
```

#### B. The Payload
Contains the **claims**—statements about an entity (typically, the user) and additional metadata. 
* **Registered/Standard Claims:** Recommended, predefined claims like `iss` (issuer), `exp` (expiration time), `sub` (subject/user ID), `aud` (audience), `nbf` (not before), and `jti` (JWT ID).
* **Public/Private Claims:** Custom claims created to share information (e.g., `roles`, `tenant_id`).
```json
{
  "sub": "usr_948271a",
  "name": "Jane Doe",
  "role": "admin",
  "jti": "uuid-8f92-411a",
  "exp": 1811634800
}
```

#### C. The Signature
The signature is used to verify that the sender of the JWT is who it claims to be and to ensure that the message wasn't altered along the way.
It is created by signing the encoded header, encoded payload, a secret key, and the algorithm specified in the header.
```javascript
// HMAC-SHA256 Signature construction
const signature = HMAC_SHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secretKey
);
```

### Critical Concept: Base64URL Encoding vs. Encryption
* **Encoding (Base64URL):** Standard JWTs (specifically JWS - JSON Web Signature) are **encoded**, NOT encrypted. Base64URL is standard Base64 modified to be safe for URLs and HTTP headers:
  * It replaces `+` with `-` and `/` with `_`.
  * It removes trailing padding characters (`=`).
  * **Anyone can decode a standard JWT** to read the payload (e.g., using [jwt.io](https://jwt.io)).
* **Signing:** Guarantees **integrity** and **authenticity**. It proves that the payload was generated by someone holding the private key/secret, and that the payload has not been modified. It does NOT guarantee **confidentiality**.
* **Encryption (JWE - JSON Web Encryption):** If you must store sensitive data (e.g., PII like social security numbers) inside the token, you must use **JWE**, which encrypts the payload, making it unreadable without the decryption key.

### Signing Algorithms: Symmetric vs. Asymmetric

```
Symmetric (HS256):
┌──────────────┐         ┌──────────────────────────────┐         ┌──────────────┐
│ Auth Service │ ──JWT──>│ Internal Service A (Verifies)│ ──JWT──>│  Service B   │
└──────────────┘         └──────────────────────────────┘         └──────────────┘
  Holds: SECRET            Holds: Same SECRET                       Holds: Same SECRET

Asymmetric (RS256):
┌──────────────┐         ┌──────────────────────────────┐         ┌──────────────┐
│ Auth Service │ ──JWT──>│ Gateway / Internal Service   │ ──JWT──>│  Service B   │
└──────────────┘         └──────────────────────────────┘         └──────────────┘
  Holds: PRIVATE Key       Holds: PUBLIC Key (via JWKS)             Holds: PUBLIC Key
```

#### 1. Symmetric (e.g., HS256 - HMAC using SHA-256)
* A single shared secret key is used to both **sign** and **verify** the token.
* **Pros:** Extremely fast computation, simple configuration.
* **Cons:** Security boundary risk. If you have a microservice system, every service that needs to verify the token must have the shared secret. If *one* microservice is compromised, an attacker gains the secret and can forge valid JWTs for the entire system.

#### 2. Asymmetric (e.g., RS256/ES256 - RSA/ECDSA using SHA-256)
* Uses a public/private key pair. The Auth Service holds the **Private Key** (used to sign tokens). All other downstream services, clients, or API Gateways hold the **Public Key** (used only to verify tokens).
* **Pros:** High security. Downstream microservices can verify tokens without any risk of forging them.
* **Cons:** Computationally slower than symmetric verification. Requires managing a key rotation infrastructure.
* **Enterprise Pattern (JWKS):** Verifying services fetch the public key dynamically from the Auth server's JWKS (JSON Web Key Set) endpoint (e.g., `https://auth.company.com/.well-known/jwks.json`). Keys are cached locally by services with a TTL (e.g., 24 hours) to eliminate network latency on every request.

## 2. Threat Modeling: Stolen Access Tokens

If an attacker steals a user's access token, they can impersonate that user until the token expires. Understanding how they get stolen and where they should be stored is a major interview topic.

### Attack Vectors
1. **Cross-Site Scripting (XSS):** An attacker injects malicious JavaScript into the client application. The script can access the client's local storage, session storage, or global variables, read the JWT, and transmit it to an external server.
2. **Cross-Site Request Forgery (CSRF):** If the JWT is stored in a cookie, the browser automatically sends it with every request to that domain. An attacker can host a malicious site that triggers a request to the target site, leveraging the user's active session.
3. **Man-in-the-Middle (MITM):** If TLS is misconfigured, terminated prematurely, or HTTPS isn't strictly enforced, tokens sent in the `Authorization` header can be intercepted in transit.
4. **Physical Access / Malicious Extensions:** Unauthorized browser extensions can access `localStorage` or read active document cookies if they have broad host permissions.

### Token Storage Comparison Matrix

| Storage Mechanism | XSS Protection | CSRF Protection | Implementation Complexity | Subdomain/Multi-domain Sharing |
| :--- | :--- | :--- | :--- | :--- |
| **LocalStorage / SessionStorage** | ❌ **Poor**. Fully accessible to malicious JS. |  **Excellent**. JS must manually read it and set headers; browser doesn't send it automatically. |  **Low**. Easy to write and retrieve. | ❌ Cannot share easily across subdomains (same-origin policy restriction). |
| **In-Memory (JS Variable)** |  **Good**. Scoped inside JS closures; harder to extract. Lost on refresh. |  **Excellent**. Browser does not send it automatically. | ❌ **High**. Requires implementing a "silent refresh" mechanism (polling or iframes). | ❌ Same-origin restricted. |
| **Standard Cookie** | ❌ **Poor**. Accessible via `document.cookie` in JavaScript. | ❌ **Poor**. Automatically sent by the browser. |  **Low**. |  **Good**. Can share across subdomains. |
| **HttpOnly, Secure, SameSite Cookie** |  **Excellent**. Cannot be read by JavaScript (immune to XSS extraction). | ❌ **Vulnerable (by default)**. Requires mitigation (SameSite flags + CSRF tokens). |  **Medium**. Requires server-side cookie configuration. |  **Excellent**. Can be configured for subdomains via `domain` flag. |

## 3. The "Blind" Server Problem

In a purely stateless JWT implementation, the server is "blind" between the time a token is issued and when it expires.

```
   Client (Attacker)                   API Gateway (Stateless)
         │                                       │
         ├───────── GET /api/v1/billing ────────>│ [Verifies Signature: Valid]
         │          Authorization: Bearer <JWT>  │ [Checks Expiry (exp): Valid]
         │                                       │ [Stateless check passed]
         │                                       │
         │<──────── 200 OK (Sensitive Data) ─────┤
```

### Why this occurs:
* The server **does not maintain session state** in a database or cache.
* The validation logic is purely cryptographic: `Is the signature valid?` + `Is current_time < exp?`.
* If a token is stolen, the server cannot distinguish between the legitimate owner and the attacker. Both present the identical cryptographic credential.

## 4. Enterprise Revocation Solutions

To solve the blind server problem, real-world systems layer security mechanisms to balance scalability (stateless performance) and immediate revocation (security).

### Solution 1: Short-lived Access Tokens + Long-lived Refresh Tokens
* **How it works:**
  1. The **Access Token** is short-lived (e.g., 10-15 minutes).
  2. The **Refresh Token** is long-lived (e.g., 7-30 days) and stored in a secure `HttpOnly`, `SameSite=Strict`, `Secure` cookie.
  3. When the access token expires, the client calls `/refresh` to get a new access token without prompting the user.
* **Tradeoff:** If an access token is stolen, the attack window is limited to a maximum of 15 minutes. Revoking the user's session requires revoking the refresh token, causing the access token to fail to renew once it expires.

### Solution 2: Refresh Token Rotation (RTR) & Reuse Detection
To prevent attackers from using a stolen refresh token to continuously generate new access tokens:

* **How it works:**
  1. Every time a refresh token is used to request a new access token, the auth server **invalidates** the old refresh token and issues a **new** refresh token.
  2. The server keeps a record of the "family" of refresh tokens or maps them to active sessions.
  3. **Automatic Abuse Detection:** If the server receives an request with an **already-used** refresh token, it assumes a replay attack. The server immediately invalidates the entire token family (revokes the current session and all active child sessions for that user), forcing an immediate re-login.

#### Sequence Diagram: RTR Replay Detection
```
Legitimate Client                 Auth Server                   Attacker (with Stolen RT1)
       │                              │                              │
       │─── POST /refresh (RT1) ─────>│                              │
       │                              │ [Validates RT1]              │
       │                              │ [Marks RT1 as USED]          │
       │                              │ [Generates AT2 + RT2]        │
       │<─── Returns AT2 + RT2 ───────│                              │
       │                              │                              │
       │                              │                              │─── POST /refresh (RT1) ───>
       │                              │                              │    (Replay Attempt)
       │                              │                              │
       │                              │ [Detects RT1 is USED]        │
       │                              │ [Revokes RT2 family]         │
       │                              │ [Returns 401 Unauthorized]   │
       │                              │<─────────────────────────────┤
       │                              │
  [Attempt to use AT2/RT2 fails]      │
  [Forced to re-authenticate]         │
```

### Solution 3: Token Epoch & Versioning (User-Level Epoch)
* **How it works:**
  1. On the database User model, we maintain a `token_epoch` or `jwt_version` (an integer, defaults to `1`).
  2. This `token_epoch` is embedded inside the JWT payload as a claim: `"epoch": 1`.
  3. During request verification, the server verifies the signature, then loads the user's record from a cache (like Redis) or database, and asserts that `jwt.epoch === db.user.token_epoch`.
* **Revocation:** To invalidate all tokens for a specific user (e.g., on password reset, security breach, or clicking "Log out of all devices"), increment the user's `token_epoch` in the database: `token_epoch++`.
* **Tradeoff:** It introduces a lookup per request. However, caching the `token_epoch` in Redis under a fast key-value store (e.g., `user:uid:epoch`) limits lookup time to < 1ms, preserving high scalability.

### Solution 4: Distributed Blacklisting with JTI (Bloom Filters)
* **How it works:**
  1. Every issued JWT contains a unique identifier claim: `jti` (JWT ID).
  2. When a user logs out or a token is revoked, the `jti` is stored in a Redis blacklist.
  3. The TTL of the blacklisted key in Redis is set to the **exact remaining time** until the token naturally expires (`exp - current_time`).
  4. On every API request, the gateway checks if the incoming token's `jti` is present in the Redis blacklist.
* **Memory Optimization (Bloom Filters) at scale:**
  Checking a database/cache for every request can create a high-throughput bottleneck. At massive scale (e.g., Netflix), a **Bloom Filter** is loaded in the memory of the API Gateway:
  * A Bloom Filter is a space-efficient probabilistic data structure.
  * It can query whether a `jti` is part of the blacklist.
  * **No False Negatives:** If the Bloom Filter says "Not Blacklisted," it is guaranteed to be correct. The request proceeds statelessly.
  * **Possible False Positives:** If it says "Blacklisted," it might be a false positive. In this case, the gateway falls back to query the actual Redis blacklist store to confirm. This saves 99% of external cache lookups.

### Solution 5: Stateful Session Reference (Hybrid Model)
* **How it works:**
  1. The client is issued a JWT containing user claims and a `session_id`.
  2. A matching session record is stored in a Redis cluster: `session:<session_id> -> { status: "active", device: "mobile" }`.
  3. The backend validates the JWT signature statelessly, then does a fast key read in Redis to verify that the `session_id` is active.
* **Pros:** Immediate session revocation, device-specific logouts (e.g., revoke only the mobile session while keeping the desktop session active), and session listing capabilities.
* **Cons:** Loses pure statelessness, but Redis scaling is a mature engineering pattern that handles millions of reads/sec easily.

## 5. FAANG Interview Q&A (Senior Engineer Level)

### Q1: JWT is encoded, not encrypted. If anyone can paste it into jwt.io and see the payload, why is it considered secure?
**Answer:**
Security is not just about **Confidentiality** (hiding data); it is also about **Integrity** (verifying the data hasn't been modified) and **Authenticity** (verifying who created it).

Standard JWTs (JWS) prioritize integrity and authenticity. The security relies on the **Signature**. The payload can be read by anyone, but if an attacker attempts to modify the payload (e.g., changing `"role": "user"` to `"role": "admin"`), the signature check will fail because the attacker does not have the server's private key or secret. Therefore, you should **never store sensitive data** (like passwords, PII, or API keys) in a standard JWT payload. If confidentiality is required, you must wrap the payload in a **JWE (JSON Web Encryption)** structure.

---

### Q2: How do you revoke a stateless JWT immediately in a distributed microservices architecture?
**Answer:**
There is a fundamental tradeoff between immediate revocation and pure statelessness. To handle immediate revocation, we implement one of three architectures depending on scaling needs:

1. **Distributed Blacklist via Redis with JTI:** The authorization service publishes token revocations to a Redis cache. Downstream microservices check incoming `jti` headers against Redis. To prevent database checks from bottlenecking microservices, we cache blacklisted IDs in the microservices' local memory or use an API gateway filter.
2. **Push Model (Event-Driven Webhooks/Pub-Sub):** When a token is revoked, the Auth Service publishes a `TokenRevoked` event to a message broker (e.g., Kafka, RabbitMQ). Downstream microservices subscribe to this topic and maintain a local in-memory blacklist of revoked `jti`s. This retains microservice autonomy and sub-millisecond local validation, with a minor event-propagation latency (typically < 100ms).
3. **Aggressive Expiry (Short TTL):** Access tokens are given a very short lifespan (e.g., 2 minutes). The system accepts the propagation delay; a compromised token is only valid for a maximum of 2 minutes before the client must hit the auth gateway to refresh, where validation occurs.

### Q3: What is the vulnerability of using symmetric HS256 in a large microservices system with 50+ services?
**Answer:**
Symmetric signing (HS256) requires all 50+ microservices to share the **exact same secret key** to verify JWT signatures. This violates the security principle of **Least Privilege**:

1. **Increased Attack Surface:** If any single microservice is compromised (e.g., through remote code execution or developer credential leakage), the entire system is breached. The attacker can extract the secret key and sign new tokens with arbitrary payloads, impersonating any user (including admins) across all other microservices.
2. **Key Rotation Complexity:** Rotating a shared symmetric key in production requires redeploying or updating configuration across all 50+ services simultaneously. If a service fails to update, it will reject valid tokens, causing system outage.
3. **Solution:** Transition to **RS256/ES256**. Only the Auth Service requires the private key to sign. The 50+ services only store or fetch the public key (via a JWKS endpoint) to verify signatures, minimizing the impact of a compromised downstream service.

---

### Q4: How does CSRF protection work when storing JWTs in an HttpOnly cookie?
**Answer:**
If a JWT is stored in an `HttpOnly` cookie, the browser automatically appends it to all requests sent to the issuing domain. This introduces susceptibility to Cross-Site Request Forgery (CSRF). To mitigate this:

1. **SameSite Cookie Attribute:** Configure cookies with `SameSite=Lax` or `SameSite=Strict`. This prevents the browser from sending the cookie on cross-site requests (e.g., clicking a malicious link on an external site).
2. **Double-Submit Cookie Pattern (Stateless):** 
   * When the user logs in, the server generates a random, cryptographically secure value (a CSRF token) and sets it as a standard (non-HttpOnly) cookie (e.g., `csrf_token`).
   * The client-side application (React/Angular) reads this cookie using JavaScript and copies the value into a custom request header (e.g., `X-XSRF-TOKEN`) for every API call.
   * The server compares the cookie value with the request header value. Since cross-origin malicious scripts cannot read cookies due to the Same-Origin Policy, they cannot populate the custom header, preventing the attack.
3. **Custom Headers:** Custom headers (like `Authorization: Bearer <token>`) are inherently immune to CSRF because browsers do not automatically construct or send custom headers on cross-site form submissions or links.

### Q5: How do you prevent replay attacks using the `jti` claim?
**Answer:**
A Replay Attack occurs when an attacker captures a valid request and resends it to the server. The `jti` (JWT ID) claim acts as a cryptographic **nonce** (number used once).

1. **JTI Assignment:** When generating the JWT, the auth server embeds a cryptographically unique identifier (e.g., UUIDv4) in the `jti` claim.
2. **Verification State:** The consuming server stores all processed `jti` values in a high-speed cache (like Redis) for the duration of the token's remaining lifespan (`exp`).
3. **Validation Check:** Upon receiving a request, the server extracts the `jti`. If the `jti` is already present in the cache, the request is rejected as a replay attack. Once the token expires naturally (`current_time > exp`), the `jti` can be safely evicted from the cache, preventing unbounded memory growth.

---

### Q6: What is a "None" Algorithm attack, and how do modern libraries protect against it?
**Answer:**
The `None` algorithm is a fallback option in the JWT specification intended for debugging. It signifies that the token is unsigned, leaving the signature section empty: `Header.Payload.`.

An attacker can exploit poorly configured JWT validation libraries by modifying a legitimate token's header to `"alg": "none"` and altering the payload (e.g., setting admin permissions). If the server library validates the token by reading the algorithm from the header and executes:
```javascript
if (header.alg === 'none') {
  return true; // Skip signature verification
}
```
The server will trust the forged payload.

**Mitigation:**
Modern validation libraries prevent this by forcing the developer to explicitly pass the expected signing algorithm to the validation function (e.g., `jwt.verify(token, secret, { algorithms: ['RS256'] })`). Any token containing an algorithm not present in the allowed list (including `none`) is rejected.

---

### Q7: Explain Token Bloat and how it impacts network performance at scale.
**Answer:**
Token Bloat refers to JWTs growing excessively large due to storing too many claims (roles, permissions, user profile details) in the payload.

**Impacts:**
1. **Network Overhead:** Because HTTP is stateless, the client sends the JWT with **every single request**. If the JWT is 4KB and the API receives 10,000 requests/sec, the network overhead is 40MB/s purely from transmission of authentication metadata.
2. **Gateway Buffer Limits:** Web servers (like Nginx) and API Gateways have strict limits on HTTP header sizes (typically 4KB to 8KB). If a JWT exceeds this limit, the gateway rejects the request with a `413 Request Entity Too Large` or `400 Bad Request` before it reaches the backend.

**Solutions:**
* Store only minimal identifiers (e.g., `sub` and `session_id`) in the JWT.
* Fetch granular permissions and user profiles downstream from a fast cache (Redis) or use a reverse proxy to strip/exchange the thick token for a thin header before routing to internal services.
